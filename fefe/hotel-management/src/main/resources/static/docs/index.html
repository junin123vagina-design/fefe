2. Desenvolvimento

2.1 Processo de Software

O projeto seguirá uma metodologia ágil, preferencialmente Kanban, para garantir entregas rápidas e adaptabilidade às prioridades do negócio hoteleiro.

2.1.1 Análise de Requisitos

O levantamento de requisitos focou na simplicidade e utilidade para o ambiente hoteleiro, evitando complexidades desnecessárias e garantindo que as funcionalidades essenciais para o controle hoteleiro sejam atendidas.

Requisitos Funcionais
CÓDIGODESCRIÇÃO
RF.001Permitir o cadastro, edição e exclusão de Clientes com validação de campos obrigatórios.
RF.002O sistema deve controlar a disponibilidade dos quartos em tempo real e evitar reservas duplicadas para a mesma data.
RF.003O sistema deve registrar o Check-in e o Check-out do cliente, atualizando o status do quarto.
RF.004O sistema deve permitir o registro de serviços adicionais consumidos pelo cliente durante a hospedagem.
RF.005O sistema deve gerar relatórios mensais de ocupação, faturamento e serviços adicionais.
RF.006Possibilitar a integração com o site existente para recebimento de novas reservas online.
Requisitos Não-Funcionais
CÓDIGODESCRIÇÃO
RNF.001Usabilidade: A interface deve ser responsiva (desktop, tablets e smartphones).
RNF.002Desempenho: O sistema deve suportar um tempo de resposta máximo de 3 segundos para consultas e registros.
RNF.003Segurança: O acesso ao sistema deve ter autenticação de usuários (Recepcionista, Gerente) e níveis de permissão.
RNF.004Confiabilidade: O sistema deve realizar backup automático diário do banco de dados.

2.1.2 Projeto

2.1.2.1 Diagrama de Casos de Uso

O Diagrama de Casos de Uso (DCU) modela as interações primárias no sistema, garantindo que as funcionalidades mais críticas (reserva, check-in, faturamento) estejam bem definidas.

Quem são os atores?

Gerente Administrativo (G)
Acesso total, configurações e relatórios estratégicos.
Recepcionista (R)
Cadastro de clientes, reserva, check-in/check-out e serviços.
Cliente (C)
Realiza reservas via integração com site.
